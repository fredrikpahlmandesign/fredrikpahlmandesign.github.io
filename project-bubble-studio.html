Bubble Studio is a generative asset generator — a tool for producing iridescent, physically-grounded soap bubble visuals, controllable through a parameter set and exportable as static images or video loops.

But the project is equally about how it was made. It started as a question: what happens when a designer with limited shader experience uses AI as a close collaborator to build something technically complex? The answer turned into a working tool I actually use.

Generated Assets

Each render is produced from the same underlying system, with different parameter combinations controlling bubble count, thickness, iridescence, flow behaviour, and surface-tension wobble. The visual language is consistent but the outputs vary — more like a material than a fixed image.

Animated Loops

The renderer runs in real-time at 30fps. Surface-tension wobble, thin-film colour flow, and bubble physics are all animated — making the exports feel alive without scripted choreography.

How it was made

I don't have a background in GLSL or GPU programming. The shader that drives this — physically-based thin-film interference, surface-tension wobble modes, animated flow fields — is genuinely complex code I could not have written alone.

The experiment was treating Claude as a technical pair: I described the visual system I wanted, asked questions when I didn't understand what the code was doing, and iterated on the behaviour by specifying it in design terms rather than shader terms. Over time the boundary blurred — I started understanding more of the code, and the AI started anticipating the design directions I cared about.

Bubble Studio became a test of that workflow. The result is a tool I designed, a shader I can read and modify, and a method I've since carried into other projects.

What I took from this

01
Designing the vocabulary matters more than the code

The best sessions happened when I was precise about visual language — "more Plateau's law at the contact point", "the colour should feel like oil on water at 30° light" — rather than asking for code changes. Descriptive precision replaced technical fluency.

02
Tools you build yourself fit differently

Because I specified every behaviour, I understand the parameter space in a way I wouldn't with off-the-shelf software. The tool is opinionated in the right directions for how I actually work. That's a side effect of building rather than buying.

03
A workflow experiment that became a repeating method

I came into this asking whether the approach was viable. The answer was yes — and I've since used variants of the same method for other tooling, generative experiments, and design systems work. The experiment succeeded, and the habit stuck.
